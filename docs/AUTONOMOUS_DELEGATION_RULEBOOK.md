## 1. 介入最小化の数値目標（運用 KPI）

「ちゃんと自律委譲できているか」の判断基準は次の 3 つの数値で測る。

| 指標 | 目標値 | 説明 |
|---|---|---|
| **A. Gonti さん総介入時間** | 30 分以内 | Phase 0（前提整備）+ 完了確認の合計時間 |
| **B. 実行中の介入回数** | 0〜3 回 | 実装フェーズ中に Gonti さんが何かを判断・操作した回数 |
| **C. 戦略判断 escalation 回数** | 0〜1 回 | Claude Code が方針 D で対応しきれず人間判断を求めた回数 |

**観察値（2026-05-04 の実績）**:
- workflow 移植プロジェクト: A=20 分、B=3 回（CMEMS 値投入、import 修正待ち、SSH 切断対応）、C=1 回（Option A vs B）
- UI 改修 Phase 1: A=12 分、B=1 回（プロンプト貼付）、C=0 回 ← **理想的**

**目標達成のセルフチェック**:
- [ ] Phase 0 を計画書 §0 に列挙し、ユーザーが一気に消化できる形にしたか
- [ ] 実行フェーズで「途中で確認」を要求する計画になっていないか
- [ ] 仕様の曖昧点を Cowork 側で潰したか（§9 参照）
- [ ] 失敗時方針が D（自己回復）になっているか

## 2. 役割分担マトリクス

| アクター | できる | できない（明示的に避ける） |
|---|---|---|
| **🧑 Gonti さん（人間）** | ブラウザ操作、SSH passphrase 入力、戦略判断、最終承認、外付けデバイス操作 | 大量の手動コマンド実行、長時間モニタリング、繰り返し作業 |
| **🤖 Cowork Claude（ノートPC）** | 計画立案、現状分析、ファイル調査（Windows 側）、計画書作成、Supervisor プロンプト生成、知見蓄積、判断支援 | sandbox-01 ファイル直接操作、git push（最終的には PowerShell 経由）、workflow 発火、ブラウザ自動化 |
| **🐧 Claude Code（sandbox-01）** | 計画書実行、Task tool で並列ワーカー起動、git 操作、gh CLI、scan_3layer.sh、自己回復ループ | ブラウザ自動化、SSH 鍵 passphrase 入力、人間裁量の戦略判断 |
| **⚙️ CI / GitHub Actions** | スケジュール実行、workflow_run 連鎖、Pages 自動デプロイ | 戦略判断、新規ロジックの発明 |

**境界線の鉄則**:
- 上の表で「できない」と書いた行に作業を振らない
- 1 ステップに複数アクターをまたぐ作業は避ける（境界が事故を生む）
- Cowork → Claude Code への引渡しは **必ず GitHub 経由**（push → pull）。直接ファイル転送は避ける

## 3. プロジェクトの 4 フェーズ標準フロー

```
[フェーズ 1: 設計]            [フェーズ 2: 前提整備]      [フェーズ 3: 自律実装]      [フェーズ 4: 完了確認]
   Cowork Claude            Gonti さん              Claude Code (sandbox-01)        Gonti さん
       ↓                        ↓                          ↓                            ↓
 計画書 + Supervisor       Phase 0 の各 G-x 項目を    Wave 1〜N を方針 D で          公開ページ・成果物
 プロンプト出力             一括消化                    自走ループ                    の目視確認
       │                        │                          │                            │
   30〜60 分                 5〜15 分                   30 分〜数時間                   5〜10 分
   （対話的）                （シーケンシャル）           （ノータッチ）                  （任意のタイミング）
```

**各フェーズの責任移譲タイミング**:

1. **設計 → 前提整備**: 計画書 §0 が完成し、Gonti さんに渡された瞬間
2. **前提整備 → 自律実装**: Phase 0 全項目に ☑ が付き、Supervisor プロンプトが Claude Code に投入された瞬間
3. **自律実装 → 完了確認**: Claude Code が「最終レポート」を出力した瞬間
4. **完了確認 → アーカイブ**: Gonti さんが目視確認 OK を出した瞬間

各タイミングで責任が「次のアクター」に **完全に渡る**。前のアクターは観察役に徹する。

## 4. Phase 0 設計の鉄則

### 4-1. なぜ Phase 0 を冒頭に集約するか

実行中に「あ、Secrets 設定してなかった」「鍵をロードしてなかった」が発覚すると、Claude Code が止まる → Gonti さんを呼ぶ → 流れ分断 → ストレス。**全部前倒しすれば実行は完全に走る**。

### 4-2. Phase 0 に入れるべきもの（チェックリスト）

#### 必ず確認する共通項目（プロジェクトに関わらず）

- [ ] sandbox-01 にログインできること（`ssh sandbox-01`）
- [ ] 個人鍵 `~/.ssh/id_ed25519_github` の存在確認
- [ ] ssh-agent に個人鍵をロード（push が必要なら）
- [ ] 対象 repo を `~/workspace/` に clone or pull
- [ ] gh CLI 認証済み（`gh auth status` が `✓ Logged in`）
- [ ] Claude Code v2.1.x 以上が利用可能（`claude --version`）

#### プロジェクトに応じて追加する項目

- [ ] **GitHub Secrets**: `gh secret list -R supergonti/<repo>` で必要なものが揃っているか確認、無ければ Web UI から投入（§7 §6-4-1 参照）
- [ ] **旧リポジトリ参照用 clone**: `git clone https://github.com/supergonti/<old-repo>.git ~/workspace/<old-repo>`
- [ ] **必要な credentials の手元保管確認**: 値が必要な Secrets について、パスワードマネージャ or .env 等から取り出せるか確認
- [ ] **tmux/screen でのセッション保護**: 1 時間以上の自走見込み時（§8-1 参照）
- [ ] **バックアップ**: 重大変更前は Timeshift snapshot や rsync 取得

### 4-3. Phase 0 各項目のフォーマット

```markdown
### G-N. <項目名>

**目的**: なぜ必要か（1〜2 行）

**実行**:

🐧 **ノートPCでSSHで実行**

```bash
コマンド例
```

**確認**:

期待される出力 or 確認方法

**NG 時**: どう対処するか

**完了マーカー**: ☐
```

### 4-4. Phase 0 完了の判定

すべての ☐ にチェック → Gonti さんが Claude Code に「Phase 0 完了。実装に進んでください」と告げる、または Supervisor プロンプトを貼り付けたら自動的に実装フェーズに突入する設計にする。

### 4-5. Phase 0 を「圧縮」するコツ

- gh CLI 認証は最初の Phase 0 で 1 回だけ。以後のセッションは `gh auth status` で再利用確認のみ
- 個人鍵のロード / 削除は標準的な「実行前 add / 実行後 delete」スタイルでテンプレ化
- Web UI 操作はブラウザ前提だが、URL を直接記載すれば 1 クリックで開ける
- 同じプロジェクトの後続セッションでは、Phase 0 の多くは「前回セッション末で完了済」になる。**「直前の状態を引き継ぐ前提」をプロンプトに書く** と楽

## 5. Supervisor プロンプトの設計鉄則

### 5-1. プロンプトは「自己完結」でなければならない

Claude Code に投げるプロンプトは、それ単体で **追加質問なしに完遂可能** であるべき。途中で「これどうします？」と聞かれる時点で設計失敗。

### 5-2. 必須セクション（テンプレートは §11）

1. **Role 宣言**: 「あなたは ... の指揮役です」
2. **前提状態の要約**: 前セッションでどこまで進んでいるか、現在のリポ HEAD など
3. **必読資料リスト**: 順序付き、数を最小限に
4. **失敗時の方針**: A/B/C/D を明示、推奨は D
5. **作業の分解（Wave 設計）**: 並列 / シーケンシャル を明記
6. **タスク詳細**: 各 Worker に何をさせるか具体的に
7. **検証フェーズ**: どう確認するか（curl / gh run list / etc）
8. **最終レポート要件**: Claude Code が何を出力すべきか
9. **片付け指示**: ssh-add -d 等を含む
10. **仕様で迷ったときの判断基準**: 曖昧排除（§9）

### 5-3. プロンプトに含めない方が良いもの

- 「臨機応変に判断してください」← 方針が無い指示は事故の元
- 「何かあれば質問してください」← Claude Code が止まる隙を生む（方針 D に反する）
- 矛盾する指示（例: 方針 D と「全部承認を取る」が同居）
- 過度に長い参考資料の本文（リンク or ファイルパスで指す）

### 5-4. Wave 設計の原則

**並列（同時起動）**:
- 別ファイルを編集するタスク
- 互いの成果物に依存しないタスク
- 検証コマンドが独立しているタスク

**シーケンシャル（順次起動）**:
- 同じファイルを編集するタスク
- 前の Wave の成果が次の Wave の入力になる場合
- リソース競合（git push 等）が起きる場合

**判定の目安**: 並列度は 4 が上限の目安（Task tool の負荷バランス）。

### 5-5. プロンプト長の目安

- 短プロジェクト（1〜3 ファイル変更）: 50〜150 行
- 中プロジェクト（4〜10 ファイル）: 150〜400 行
- 大プロジェクト: 計画書を別ファイル化し、プロンプトでは「計画書を読んで」+ 重要要素のみ抜粋

## 6. 失敗時の方針 A/B/C/D 使い分け

| 方針 | リトライ | 推奨シーン |
|---|---|---|
| A: 保守的 | 0 回（即ロールバック）| 本番影響が極めて大きい変更 |
| B: 積極的 | N 回試行 | 通常運用でリスク中程度 |
| C: テスト的 | 0 回（停止+報告）| 初回テスト・新パイプライン検証 |
| D: 自己回復（**標準**）| 同一エラー最大 5 回まで | 通常の自律委譲 |

**標準は方針 D**。理由：
- 人間が逐次確認すると、PowerShell ↔ SSH 取り違え等の事故が起きる（L-008/L-013）
- 反復的な小バグ（cache:pip / 相対 import など）は AI のほうが速く確実
- 戦略判断（Option A vs B 等）が必要なときだけ停止して escalation

**方針 D が止まるべき条件**:
1. 戦略的判断（仕様の解釈が複数ある等）
2. N 回試行しても解決しない
3. ブラウザ操作・SSH 鍵 passphrase など人間限定作業が必要
4. 想定外の重大状態（巨大 diff、別ブランチへの誤 push、master 履歴破壊）

## 7. アンチパターン集（やってはいけないこと）

### 7-1. プロンプト・指示文での反パターン

| アンチパターン | なぜ NG | 正解 |
|---|---|---|
| プロンプト文字列をコードブロック内に書く（例: `PS C:\Users\super>` を ``` ``` の中に） | 初心者がコマンドだと勘違いして実行しエラー（L-010） | ラベルはコードブロック外に太字 + 絵文字で表示 |
| 複数コマンドを 1 つのコードブロックにまとめる | PowerShell で最後だけ Enter 待ちになる事故（L-011） | **1 コマンド 1 ブロック** |
| 環境ラベルを書かない | ノートPC か SSH か取り違えて bash 構文エラー（L-013） | 必ずラベル明記 |
| 「臨機応変に」「何かあれば質問」 | 方針 D に反し、Claude Code が止まる | 方針を明示 |
| Phase 0 を実行中に分散させる | ストレス源、流れ分断 | 必ず冒頭に集約 |

### 7-2. 運用・インフラでの反パターン

| アンチパターン | なぜ NG | 正解 |
|---|---|---|
| 個人鍵を ssh-agent に常駐 | Phase 2 鍵分離の趣旨に反する | 必要時のみ add、完了後 delete |
| Secrets を CLI で `--body "<値>"` 渡し | シェル履歴に値が残る | Web UI 投入 or 対話入力 |
| `cache: 'pip'` を入れたが requirements.txt 無し | setup-python が失敗（L-015） | 空でも requirements.txt を置くか cache 行を消す |
| `python -m pkg.module` で `from xxx import` が絶対 import | ModuleNotFoundError（L-017） | `from .xxx import` の相対 import に |
| 長時間 Claude Code 自走中に SSH 切断対策なし | プロセス終了 | tmux/screen でセッション保護 |
| Cowork サンドボックス bash で `git status` 確認 | core.autocrlf 未設定で全行差分検知（L-009） | 確認は Windows 側で行う |

### 7-3. ファイル受け渡しでの反パターン

| アンチパターン | なぜ NG | 正解 |
|---|---|---|
| Cowork ローカル → sandbox-01 へ scp / rsync | Phase 2 鍵分離との整合性が悪い、混乱 | **GitHub 経由（push → pull）が原則** |
| ローカル変更をたくさん溜めて push を後回し | sandbox-01 が古い情報で動く / コンフリクト懸念 | 計画書ができた段階で即 push |
| `git add -A` を Cowork サンドボックス bash で | CRLF 差分を巻き込む可能性 | Windows PowerShell で個別 `git add <path>` |

## 8. インシデント時の対処

### 8-1. SSH 切断（Claude Code 自走中）

**現象**: `client_loop: send disconnect: Connection reset` が出る、PowerShell に戻る

**対処**:
1. 慌てない。git は完了済コミットを残す
2. SSH 再接続: `ssh sandbox-01`
3. 状態確認: `cd ~/workspace/<repo> && git status && git log --oneline -10 && ssh-add -l`
4. 完了済 → 公開ページ確認のみ
5. 未完了 → Claude Code を再起動して「resume プロンプト」を投げる（直近の状態を要約 + 残作業）

**予防**: 
- 1 時間以上の自走見込み時は **`tmux new -s claude` でセッション保護してから claude を起動**
- 切断しても tmux セッションは生き続ける、`tmux attach -t claude` で再接続できる

### 8-2. Claude Code がフリーズ / 質問してくる

**症状**: 入力を促されている / 動きが止まっている

**判定**:
- ステータス行に「Honking...」「Manifesting...」等が出ている → 思考中、待つ
- 「Continue?」「choose option」等 → 介入が必要

**介入が必要な場合の標準応答**:
- 「continue」「OK」「yes」など短い応答で進められる場合が多い
- 戦略判断が必要なら、**Cowork に状況を貼って判断を仰ぐ**
- ループに陥っているなら `Esc` で中断、状態を保存して整理

### 8-3. 戦略判断要求が来た（escalation）

**症状**: Claude Code が「Option A vs B どちらにしますか」のような選択を求めている

**対処**:
1. 内容を Cowork にそのまま貼る
2. Cowork が現状分析と推奨を返す
3. 判断を Claude Code に伝える
4. Claude Code が再開

これは **方針 D の正しい挙動**。escalation を恥じる必要はない。

### 8-4. workflow が失敗する（CI 失敗）

Claude Code は方針 D で `gh run view --log-failed` → 修正 → 再 push → 再発火 を自動化する。Gonti さんは介入不要。

## 9. 仕様曖昧排除の責務分担

### 9-1. Cowork の責務

計画書 §6 に「仕様確認事項」をチェックリスト化し、**実装に必要な数値・色・位置・順序・判定基準を全て確定** させる。曖昧なら Gonti さんに確認 Q を出す（できれば 1 回にまとめる）。

### 9-2. Gonti さんの責務

Cowork からの確認 Q に箇条書きで回答する。回答は短くて OK（「2 種以上」「紺色グラデ」など）。

### 9-3. Claude Code の責務

計画書 §6 に書かれた仕様判断に従って実装する。曖昧なら停止 → escalation。**勝手に解釈しない**。

### 9-4. 確認 Q の例（過去実績）

良い Q（明確な答えが返る）:
- 「複数魚種の判定基準は？2 種以上で OK ですか？」
- 「確定ボタンの色は？紺・緑・GitHub 風どれか」
- 「初期日付は？今日 / 釣果データの最新 / その他」

悪い Q（答えが多義的）:
- 「いい感じにしてください」← Cowork 側でデフォルト案を出す方が早い
- 「どう思いますか？」← 開かれすぎ

## 11. Supervisor プロンプト テンプレート（コピペ用、`<...>` を埋める）

```
あなたは <プロジェクト名> の指揮役です。前セッションで <前提状態> まで完了済。本プロンプトに必要情報を全て含めているので、追加質問なしで完遂してください。

【最初にやること】
1. cd ~/workspace/<repo>
2. git pull origin main で最新化
3. docs/AUTONOMOUS_DELEGATION_RULEBOOK.md を読む（特に §6 失敗時の方針、§7 アンチパターン）
4. docs/COWORK_HANDOFF_GUIDE.md を読む（特に §6-0 環境ラベル、§9 知見ログ）
